## 第一部分：不看代码复述完整链路

你必须脱离编辑器，完整讲出下面这条链路：

1. `main.cpp` 解析配置
2. 创建 `WebServer`
3. 初始化日志、数据库连接池、线程池、触发模式
4. 创建监听 socket、epoll、信号管道
5. `epoll_wait()` 等待事件
6. 新连接到来时 `accept` 并初始化 `http_conn`、定时器
7. 读事件到来，根据 Reactor 或模拟 Proactor 进入不同路径
8. `http_conn` 解析 HTTP 请求
9. 进入静态资源分支或者登录注册分支
10. 组织响应并写回客户端
11. 活跃连接刷新定时器，超时连接被回收
12. 关键链路通过日志记录

要求不是背标题，而是每一步至少能补一句“为什么”。

例如：

- 为什么信号要走管道
- 为什么要线程池
- 为什么要连接池
- 为什么返回静态资源时用 `mmap + writev`

## 第二部分：把整套系统拆成 4 条主链路

今天你必须手写这 4 条链路，不能省略函数名。

### 1. 启动链

最低要求写到：

`main -> Config::parse_arg -> WebServer::init -> log_write -> sql_pool -> thread_pool -> trig_mode -> eventListen -> eventLoop`

要达到的水平：

- 能解释每一步初始化为什么排在这个位置

### 2. 新连接接入链

最低要求写到：

`eventLoop -> dealclientdata -> accept -> timer -> http_conn::init`

要达到的水平：

- 能解释为什么接入连接时不只是 `accept`

### 3. 请求处理链

最低要求写到：

`epoll_wait -> dealwithread -> 线程池或主线程读 -> http_conn::process -> process_read -> do_request -> process_write -> write`

要达到的水平：

- 能根据并发模型切换，讲出两个版本

### 4. 超时清理链

最低要求写到：

`alarm -> 信号处理 -> 管道事件 -> dealwithsignal -> tick -> cb_func -> close_conn`

要达到的水平：

- 能说清为什么这条链路能避免长时间无效连接占资源

## 第四部分：给自己做一次面试式验收

今天必须回答下面这些问题，而且回答时要带代码路径，不允许只讲概念。

### 基础链路问题

- 服务器启动后，主线程第一个长期阻塞在哪里
- 一个新连接进入后，最先走到哪个函数
- 一次 HTTP 请求的总入口函数是什么

### 并发模型问题

- Reactor 和模拟 Proactor 的代码分叉点在哪里
- 半同步/半反应堆在这个项目里分别对应哪些代码角色
- LT 和 ET 在这个项目里体现在哪些位置

### HTTP 和业务问题

- GET 和 POST 从哪里开始出现不同处理
- 登录注册逻辑在什么函数里发生
- 静态资源返回为什么不需要特殊业务逻辑

### 工程性问题

- 定时器怎么挂到连接上
- 超时清理为什么不直接在信号函数里做
- 连接池和线程池分别解决什么问题
- 日志为什么要支持异步

### 评价问题

- 这个项目的 3 个优点
- 这个项目的 3 个局限
- 如果你继续改，会优先改哪 2 个点
